核心設計原則

Gen1 的 ETD 洩漏讓整個研究重新出發。Gen2 的核心問題變成：

在嚴格的 pre-departure leakage control 下，哪些特徵真正有預測力？

Tip

核心發現（12 版本歸納）：只有真正新的外部資訊才能提升 AUC；同源重編碼全部無效。

設計

V1 包含三類特徵，使用嚴格時間分割（但仍保留 ETD 做比較基準）：

  • Schedule + Calendar：出發月份、星期、時段、路線
  • Tail Chain：前一班延誤（simple one-hop）
  • DFW METAR Weather：起飛前 T-2h 的實時天氣觀測
  • ETD Snapshot（T-2h+ETD 版）：航空公司運營系統的預計起飛時間偏差

文獻參考：Zhou (2025) AUC=0.898、Berkeley (2025) F1=62.9%、Chen (2019) Tail chain、Qu (2023) CNN-MLSTM

結果

V1 最佳（T-2h+ETD + XGBoost）：AUC = 0.7623，F1 = 0.5031

特徵集 最佳 AUC
Clean（排班+日曆） 0.6226
T-24h（+尾鏈+天氣） ~0.70
T-2h（+即時天氣） ~0.72
T-2h+ETD（+運營快照） 0.7623
Warning

V1 的 ETD 問題：T-2h+ETD 版的高 AUC 源自 ETD 偏差直接反映延誤狀況。與 Gen1 的致命洩漏相同。V2 起移除 ETD。

主要結論

  • XGBoost 在所有特徵版本下均優於 RF 和 DNN
  • Tail chain（前一班延誤）貢獻了 T-24h 版本的最大 AUC 提升
  • 深度學習（DNN）在表格資料上不優於 tree-based 模型

為什麼 V2 是重要里程碑

這是唯一一次「同源但文獻驗證過的特徵工程」帶來顯著提升的案例。後續版本反覆確認：只有真正新的外部資訊才能突破 AUC 上限，V2 的 9 個特徵是邊界案例（有文獻強力支持的人工設計特徵）。

V3：Two-Stage Departure Delay Prediction

設計：Stage I 用 XGBoost 回歸器預測出發延誤，Stage II 把預測值加入特徵再預測到達延誤。

Warning

Stage I R² 僅 0.05–0.18，預測品質不足以支撐 Stage II

Stage I 低品質根本原因：出發延誤的大部分變異來自 T-2h 之後的事件（地面延誤、ATC 指令、突發天氣），這些在 T-2h 無法預知。

V3 最佳 AUC = 0.7357（T-2h+Enhanced + RF），僅比 V2 高 +0.0005，邊際到幾乎無意義。

第一次確認模式：同源 meta-feature 無效。

V4：Destination Airport METAR Weather（11 個特徵）

設計：加入 277 個目的地機場的 NOAA IEM ASOS/METAR 觀測值（T-2h）。

新特徵：dest_wx_temp_f, wind_speed_kt, wind_gust_kt, visibility_mi, precip_1hr_in, ceiling_ft, has_thunder, has_fog, has_snow, pressure_hpa, wx_severity

V4 最佳 AUC = 0.7373（T-2h+Enhanced + XGBoost），vs V2 的 0.7352，+0.0021

正向但微弱。原因：T-2h 的目的地天氣快照是單一時間點觀測，無法捕捉飛行途中的天氣動態。

V5：BTS Monthly Airport Statistics（6 個特徵）

設計：加入 BTS 目的地機場月度統計（delay rate、avg delay、delay std dev，origin/dest 各三個）。

Warning

V5 最佳 AUC = 0.7355（−0.0018 vs V4）——負向結果

月度粒度對個別航班預測太粗；dest_airport_encoded 已隱含機場層級的延誤傾向。 Coverage：204/286 目的地機場（71.3%），724,660 航班（90.1%）。

第二次確認模式：粗粒度同源資訊無效。

設計

V6 以 V4（當時最佳 AUC=0.7373）為基礎，兩步驟優化：

Step 1：SHAP 逐步後向消除

從 T-2h+Enhanced 移除 14 個低貢獻特徵，包括：dfw_avg_aadt, wx_has_snow, dest_wx_has_snow, prev_flight_landed, dest_wx_has_fog, dest_wx_has_thunder, is_weekend, dest_wx_visibility_mi, dest_wx_wind_gust_kt, prev_flight_delayed_15, wx_has_fog, dest_is_international, wx_visibility_mi, is_holiday

Step 2：Optuna Bayesian HPO（100 trials）

Optuna 最佳 Val AUC = 0.7541｜搜索時間：2310 秒（38.5 分鐘）

最佳超參數（XGBoost）：learning_rate = 0.006，max_depth = 10，gamma = 4.79，colsample_bytree = 0.33

結果

Tip

V6 最佳 AUC = 0.7420（T-2h+Enhanced + XGBoost），vs V4 的 0.7373，+0.0047

文獻依據：Borboudakis & Tsamardinos (arXiv:2410.06815) — SHAP backward elimination；Akiba et al. (2019) Optuna；Bergstra et al. (2011) HPO

V6 從此成為所有後續版本的基準。

V7：Temporal + Route Features（8 個新特徵）

新增：dest_delay_rate_7d/30d, route_daily_freq, aircraft_legs_today, dfw_delay_rate_1h/6h

Warning

V7 最佳 AUC = 0.7366（T-2h+Enhanced + XGB+CatBoost），vs V6 的 0.7420，−0.0054

12 版本中最差的單次退步。

根本原因：depth=10 的 XGBoost 已自動捕捉這些時序模式；新 8 個特徵全部冗餘。V6 的 colsample_bytree=0.33 也未針對新特徵重新優化，新特徵可能未被充分利用。

V8：SHAP Interaction Features + Focal Loss

兩個嘗試： 1. SHAP 交叉特徵（15 個，如 dep_hour × wx_severity） 2. Focal Loss 處理類別不平衡（positive rate 23.3%）

Warning

V8 最佳 AUC = 0.7407，兩者皆負

SHAP 交叉特徵：depth=10 已隱式捕捉交叉效果，手動加入冗餘。 Focal Loss：在 23.3% 不平衡程度下犧牲 AUC 換 F1（F1 提升至 0.4744，但 AUC 下降）。

V7、V8 連續兩次負向結果確認：AUC ≈ 0.74 是當前特徵空間的上限。需要全新外部資訊才能突破。

設計

V9 引入 TAF（Terminal Aerodrome Forecast） 天氣預報資料——與 V4 METAR「觀測值」的本質差異：TAF 是「預報未來抵達時的天氣狀況」。

資料來源：Iowa Environmental Mesonet (IEM) 公開存檔，212 個機場（覆蓋 89.9% 目的地）

特徵 覆蓋率
taf_dest_wind_kt 76.8%
taf_dest_gust_kt 76.8%
taf_dest_visibility_mi 85.6%
taf_dest_ceiling_ft 35.6%
taf_dest_has_precip 85.6%
taf_dfw_wind_kt 93.3%
taf_dfw_ceiling_ft 38.9%
taf_dest_ceil_delta 35.6%
taf_severity 100.0%

結果

Tip

V9 最佳 AUC = 0.7440（T-2h+Enhanced + XGBoost），vs V6 的 0.7420，+0.0020

打破 V7、V8 的負向連跌。

為什麼 TAF 有效但 METAR 有限

TAF 是預報系統輸出的「未來天氣預測」，在 T-2h 預測時就能反映預計到達時段的天氣狀況；METAR 只是「現在的天氣觀測」，對長程航班（飛行時間 > 2 小時）的目的地天氣代表性有限。

再次確認模式：帶入真正新的外部資訊（氣象預報系統的輸出）才能突破 AUC 上限。

文獻：Rebollo & Balakrishnan (2014)；Kim et al. (2016)

V10：Target Encoding + Network Weather Proxy

Target Encoding：用 sklearn TargetEncoder(cv=5) 替換 dest_airport, gate, subfleet 的 label encoding。

Warning

負向原因：XGBoost depth=10 對 label encoding 已足夠有效；TE 的優勢主要見於線性模型和淺層樹（Pargent et al. 2022）。

Network Weather Proxy：計算 9 大 Hub 機場（ATL, ORD, DEN, LAX, CLT, JFK, SFO, EWR, LAS）的平均天氣嚴重度。

Warning

V10 最佳 AUC = 0.7429（−0.0011 vs V9）

Network proxy 原因：現有 DFW 壅塞特徵已捕捉本地延誤狀態；遠端 Hub 天氣對 DFW 的影響在 T-2h 窗口內太微弱。

V11：NOTEARS DAG + 3-Model OOF Stacking

NOTEARS DAG：用連續優化學習因果圖，把同因果叢集的特徵加入 XGBoost 的 interaction_constraints。

Warning

致命問題：線性 NOTEARS 找到 0 條從任何特徵到 delayed_15 的邊。

Markov Blanket 為空，interaction_constraints 反而嚴重限制 XGBoost 學習能力。XGB_DAG 在 T-2h+Enhanced 版本中 AUC 暴跌至 0.6793（vs V9 的 0.7440，差 0.065）。

3-Model OOF Stacking：XGBoost + CatBoost + LightGBM，logistic regression meta-learner。

Warning

V11 最佳 AUC = 0.7414（−0.0026 vs V9）

三個 GBDT 模型相關性太高，Stacking 無法從互補性中獲益。LightGBM 在 meta-learner 中收到負權重。

V12：Delay-Absorption Meta-Feature

靈感來源：arXiv:2512.08197（Li 2025，AUC=0.898，但依賴出發延誤特徵）

設計：Stage I CatBoost 預測「前班延誤是否被吸收」（absorb_label），Stage II 加入 absorb_score。

Stage I OOF AUC = 0.6946（吸收標籤 absorb=1 佔 69.5%）

Warning

V12 最佳 AUC = 0.7423（−0.0017 vs V9）

Stage I 品質不足；8 個輸入特徵（turnaround_min 等）已直接存在於 V9 特徵集，Stage I 只是非線性重組而非新資訊。Li (2025) 的高 AUC 依賴出發延誤特徵（leakage），我們的嚴格版本無法複製。

V9 最終確認為 12 個版本的最佳模型（AUC = 0.7440）。

問題發現

V9 的 tail chain 使用排班週轉門檻判斷「前班是否已到達」——但這個判斷有細微 leakage 風險：在預測時刻（T-2h），部分前班實際上還沒到達，但被錯誤標記為已到。

修正

改用實際到達時間而非排班時間判斷 tail chain 可用性。

影響範圍 比例
T-2h 版本受影響航班 1.04%
T-24h 版本受影響航班 77.9%（→ T-24h 版本永久棄用）

V9c 最終 AUC = 0.7418（−0.0022 vs V9，僅 1.04% 航班有差異）

V9c 成為 Gen2 官方定稿，所有後續分析（Phase 1–4）的基礎。

V9d：Right-Censored Tail Chain（研究用）

V9d 進一步將 tail chain 從「前班已到→用實際延誤，未到→用 0」改為右刪截（Right-Censoring）方式，用 7 個特徵編碼「前班尚未到達時的已知局部資訊」（逾時分鐘數、預計剩餘時間等），NaN 表示「此時無法知道」。

V9d 時間序 AUC = 0.7291（vs V9c 0.7418，−0.0127）

隨機分割 AUC = 0.7760（+0.047），差距再次確認時間序列分割的重要性。

V9d 用於 Phase 1–4 綜合分析（43 個特徵，包含 7 個右刪截尾鏈特徵）。
